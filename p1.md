MODULE 2: PATIENT HEALTHCARE VAULT
Project: Patient-Sovereign Prescription Intelligence Network

Stack: FastAPI · Supabase · Python 3.11+

MISSION
Implement a 10-layer, defense-in-depth security architecture for the Patient Healthcare Vault. Every byte of healthcare data must be encrypted, tokenized, integrity-verified, and access-controlled before touching the database. Zero plaintext at rest. Zero trust by default.

LAYER 1 — AES-256-GCM Field Encryption
Implement a HealthcareEncryptor class using Python's cryptography library:

Algorithm: AES-256-GCM (authenticated encryption)
Generate a unique 96-bit nonce per encryption call
Return: {ciphertext, nonce, tag} as base64-encoded JSON blob
Apply to ALL fields in: medical_records, reports, prescriptions, ocr_extractions, wearable_metrics, analytics, appointments
Never store plaintext. Decrypt only on authorized retrieval.


LAYER 2 — Per-Patient Vault Keys
Implement PatientVaultKeyManager:

On patient registration, generate a unique 256-bit AES key (patient_vault_key)
Store the encrypted vault key (never raw) in patient_key_store table
Key lookup: patient_id → encrypted_vault_key
Compromise of PAT001's key must cryptographically isolate all other patients

Table: patient_key_store
- patient_id (FK)
- encrypted_vault_key (bytea)
- key_version (int)
- created_at, rotated_at

LAYER 3 — Envelope Encryption
Implement EnvelopeEncryptionService:

Master Key (KEK): Stored in environment variable / secrets manager (never in DB)
Patient Key (DEK): AES-256 key, encrypted with KEK before DB storage
Flow: plaintext → encrypt(DEK) → ciphertext; DEK → encrypt(KEK) → encrypted_DEK in DB
On retrieval: decrypt DEK using KEK → decrypt data using DEK
Implement wrap_key(dek, kek) and unwrap_key(encrypted_dek, kek)


LAYER 4 — Medical Data Tokenization
Implement MedicalTokenizer:

Sensitive diagnosis/medication values → opaque tokens (MED_XXXXXXXX)
Token format: MED_ + 6-char alphanumeric (collision-resistant, non-reversible without vault)
Token vault table:

Table: token_vault
- token (varchar PK)
- encrypted_value (bytea)  ← encrypted with master key
- data_category (enum: diagnosis, medication, condition, procedure)
- created_at

Expose: tokenize(value) → token, detokenize(token, requester_id) → value (with audit)


LAYER 5 — Supabase Row Level Security
Implement RLS policies for all healthcare tables. Enable RLS on every table. Roles: patient, doctor, admin, super_admin.
sql-- patients table
CREATE POLICY "patient_self_access" ON patients
  FOR ALL USING (auth.uid() = user_id);

-- medical_records
CREATE POLICY "doctor_authorized_access" ON medical_records
  FOR SELECT USING (
    auth.jwt()->>'role' = 'doctor'
    AND patient_id IN (
      SELECT patient_id FROM doctor_patient_assignments
      WHERE doctor_id = auth.uid() AND active = true
    )
  );

CREATE POLICY "admin_hospital_scope" ON medical_records
  FOR ALL USING (
    auth.jwt()->>'role' = 'admin'
    AND hospital_id = (auth.jwt()->>'hospital_id')::uuid
  );

CREATE POLICY "super_admin_all" ON medical_records
  FOR ALL USING (auth.jwt()->>'role' = 'super_admin');
Generate equivalent policies for: reports, prescriptions, appointments, wearable_metrics, ocr_extractions, analytics.

LAYER 6 — JWT Security
Implement JWTSecurityService:

Token signing: RS256 (asymmetric, private key signs, public key verifies)
Claims required: sub, role, hospital_id, patient_id (if patient), permissions[], consent_scope[], iat, exp (15min access / 7d refresh)
Implement create_token(payload), verify_token(token), refresh_token(refresh_token)
Middleware: JWTMiddleware — validate on every request, attach decoded claims to request.state
Reject: expired, tampered, missing required claims, wrong algorithm


LAYER 7 — Consent-Based Decryption Gate
Implement ConsentEnforcementService:

Before ANY decryption: call verify_consent(patient_id, requester_id, data_type)
Consent stored in Module 1's consent registry (assume REST endpoint available)
If consent not granted → raise ConsentDeniedException (HTTP 403), log attempt
Consent is per-data-category: [reports, prescriptions, wearable, analytics]
No consent bypass under any role — even super_admin requires audit-logged override with reason


LAYER 8 — Immutable Audit Logging
Implement AuditLogger with tamper-resistant log chain:
pythonclass AuditEvent(BaseModel):
    event_id: UUID
    timestamp: datetime
    actor_id: UUID
    actor_role: str
    patient_id: UUID
    action: Literal["VIEW","DOWNLOAD","UPLOAD","UPDATE","DELETE","DECRYPT","KEY_ROTATION","CONSENT_CHECK"]
    resource_type: str
    resource_id: UUID
    ip_address: str
    success: bool
    previous_hash: str   # hash of prior audit row
    event_hash: str      # SHA-256(this row content + previous_hash)

Each audit record hashes the previous record → tamper-evident chain
Write to append-only audit_log table (no UPDATE/DELETE policies)
Expose: log_event(...), verify_chain(from_id, to_id)


LAYER 9 — SHA-256 Integrity Verification
Implement IntegrityVerifier:

On upload: SHA-256(plaintext_content) → stored in content_hash field alongside encrypted blob
On retrieval: decrypt → SHA-256(decrypted) → compare with stored hash
Mismatch → raise TamperingDetectedException, log critical audit event, return 422
Apply to: all reports, prescriptions, medical files

FILE STORAGE SECURITY
Implement SecureFileStorageService:
Upload Flow:
raw_file → SHA256(raw) → AES-256-GCM encrypt(patient_DEK) → store blob in Supabase Storage
→ DB row: {patient_id, file_type, encrypted_blob_path, content_hash, nonce, size, uploaded_by}

Download Flow:
fetch encrypted_blob → verify content_hash → decrypt(patient_DEK) → stream to requester

Never expose raw file URLs
Generate signed, time-limited (15min) download tokens for authorized access
File metadata in DB; encrypted blob in private Supabase bucket (no public access)
